# BoundaryML Positioning

## One-liner

BoundaryML is a modeling and governance layer for human-AI workflows.

## Short Pitch

BoundaryML turns vague, AI-assisted work into structured execution kits with clear human-AI boundaries, required context, output contracts, review gates, acceptance criteria, and downstream handoff rules.

Its first focused use case is Agentic Development: preparing software development work before Coding Agents such as Codex, Claude Code, GitHub Copilot, Cursor, Hermes, OpenClaw, or similar tools execute.

## Long Pitch

AI is entering complex workflows across software development, content production, operations, data analysis, consulting delivery, enterprise approval, and automation. The core problem is no longer only whether a team uses AI, but whether the team can define which nodes should be done by humans, which nodes can be done by AI, where human-AI collaboration is required, and which outputs must pass review before entering downstream execution.

BoundaryML solves this upstream planning and governance problem. It turns vague AI-assisted work into a structured Workflow with Node Contracts, Execution Modes, Review Gates, Prompt Packs, Checklists, Artifact Contracts, Responsibility Maps, Risk Reports, and Execution Kits. The point is not to make AI execute everything; the point is to make AI-assisted work explicit, reviewable, auditable, and handoff-ready.

Agentic Development is BoundaryML's first focused use case and commercial wedge. In this scenario, BoundaryML prepares software development work before Coding Agents execute and produces an Agent-ready Execution Kit that can be reviewed by humans and then handed to Codex, Claude Code, GitHub Copilot, Cursor, Hermes, OpenClaw, or similar tools.

Agentic Development is a wedge, not the whole product boundary.

Agentic Development 是 BoundaryML 的第一个爆破点，不是 BoundaryML 的全部定义。

BoundaryML does not replace coding agents, issue trackers, documentation tools, or workflow runtimes. It creates the planning, governance, and handoff layer between vague AI-assisted work and downstream execution.

## Positioning Against Alternatives

| Alternative | What it is | BoundaryML position |
| --- | --- | --- |
| Jira / Linear | Project and issue tracking systems. | BoundaryML prepares human-AI workflow boundaries, node contracts, review gates, and handoff rules before or alongside issue creation. In Agentic Development, these boundaries can become agent-ready task packs. |
| Notion / Docs | Flexible documentation spaces. | BoundaryML turns loose docs into structured execution kits with validation, governance, output contracts, and review gates. |
| Miro / flowchart tools | Visual mapping and diagramming tools. | BoundaryML is not a generic diagramming tool; its workflow map is tied to verifiable human-AI boundaries, risk, ownership, output contracts, and exportable execution assets. |
| Codex / Claude Code / Copilot / Cursor / Hermes / OpenClaw | Coding agents and assistants that execute or assist with software work. | BoundaryML prepares Agentic Development work before these tools execute, but Coding Agents are a focused use case rather than the whole BoundaryML product boundary. |
| LangGraph / workflow runtime | Runtime framework for stateful agent or application workflows. | BoundaryML is a planning, governance, and handoff layer, not an execution runtime. |
| Dify / n8n / automation builders | AI app and automation builders. | BoundaryML defines human-AI workflow governance and execution kits rather than building automations or running production workflows. |
